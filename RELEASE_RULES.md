# Release Rules and Guidelines

## 🚀 Release Branch Policy

### Main Branch - Stable Releases Only
- **Version Format**: `v1.0.0`, `v2.1.0` (semantic versioning)
- **Release Type**: Production-ready, stable releases
- **No Beta/RC**: Never include beta, RC, or pre-release versions
- **Assets**: Only production-ready files

### Testing Branch - Beta/Pre-releases Only
- **Version Format**: `v1.0.0-beta.1`, `v2.0.0-rc.1`, `v1.2.0-alpha.3`
- **Release Type**: Beta, RC, alpha, and development releases
- **No Stable**: Never tag stable releases on testing branch
- **Assets**: Can include debug builds and test files

## 📋 Release Asset Rules

### ✅ Allowed Release Assets
1. **Binary executables** (if applicable)
   - `warp-nextdns-manager-linux`
   - `warp-nextdns-manager-windows.exe`
   - `warp-nextdns-manager-macos`

2. **Archive files**
   - `warp-nextdns-manager-v1.0.0.tar.gz`
   - `warp-nextdns-manager-v1.0.0.zip`

3. **Source code** (auto-generated by GitHub)
   - `Source code (zip)`
   - `Source code (tar.gz)`

### ❌ Forbidden Release Assets
- `*_SUMMARY.md` files
- `*.tmp`, `*.temp`, `*.bak` files
- Log files (`*.log`)
- Development/debug files
- Configuration files with secrets
- IDE-specific files
- Cache directories

## 📝 Release Notes Rules

### Release Notes MUST:
1. **Include ONLY**:
   - Version number and date
   - Changes from `CHANGELOG.md`
   - Installation instructions (brief)
   - Known issues (if any)

2. **Format Example**:
   ```markdown
   ## Version 1.0.0 - 2024-01-24
   
   ### What's Changed
   - Feature: Added new DNS configuration
   - Fix: Resolved connection timeout issue
   - Improvement: Enhanced error handling
   
   ### Installation
   Download the appropriate binary for your platform and follow the installation guide.
   
   ### Full Changelog
   See [CHANGELOG.md](CHANGELOG.md) for complete history.
   ```

### Release Notes MUST NOT:
- Include development summaries
- Contain internal notes
- Reference temporary files
- Include verbose debug information
- Contain file paths or internal structure

## 🛡️ Enforcement Mechanisms

### 1. Pre-commit Hook Enhancement
The pre-commit hook will:
- Block releases with wrong version format on wrong branch
- Prevent summary files in release commits
- Validate release tag format

### 2. GitHub Actions Validation
The release workflows will:
- Check branch before creating release
- Validate version format matches branch
- Clean release assets before upload
- Verify release notes format

### 3. Release Scripts
Located in `scripts/`:
- `create-release.py` - Automated release creation
- `validate-release.py` - Pre-release validation

## 🔧 Release Workflow

### For Stable Release (main branch):
```bash
# 1. Ensure on main branch
git checkout main

# 2. Update CHANGELOG.md
echo "## v1.0.0 - $(date +%Y-%m-%d)" >> CHANGELOG.md

# 3. Commit and tag
git add CHANGELOG.md
git commit -m "chore: prepare release v1.0.0"
git tag -a v1.0.0 -m "Release version 1.0.0"

# 4. Push
git push origin main --tags
```

### For Beta Release (testing branch):
```bash
# 1. Ensure on testing branch
git checkout testing

# 2. Update CHANGELOG.md
echo "## v1.0.0-beta.1 - $(date +%Y-%m-%d)" >> CHANGELOG.md

# 3. Commit and tag
git add CHANGELOG.md
git commit -m "chore: prepare beta release v1.0.0-beta.1"
git tag -a v1.0.0-beta.1 -m "Beta release version 1.0.0-beta.1"

# 4. Push
git push origin testing --tags
```

## 🚫 Absolute Rules

1. **NEVER** create or commit files matching:
   - `*_SUMMARY.md`
   - `*_summary.md`
   - `RELEASE_SUMMARY.md`
   - Any variation of summary files

2. **NEVER** include in releases:
   - Temporary files
   - Build artifacts not meant for distribution
   - Development-only files

3. **ALWAYS** validate before release:
   ```bash
   python3 scripts/validate-release.py
   ```

## 📊 Version Numbering

### Stable (main branch):
- **Major**: `v1.0.0` → `v2.0.0` (breaking changes)
- **Minor**: `v1.0.0` → `v1.1.0` (new features)
- **Patch**: `v1.0.0` → `v1.0.1` (bug fixes)

### Pre-release (testing branch):
- **Beta**: `v1.0.0-beta.1`, `v1.0.0-beta.2`
- **RC**: `v1.0.0-rc.1`, `v1.0.0-rc.2`
- **Alpha**: `v1.0.0-alpha.1`, `v1.0.0-alpha.2`

## 🔍 Validation Checklist

Before any release:
- [ ] Correct branch for release type
- [ ] No summary files exist
- [ ] Version format matches branch policy
- [ ] CHANGELOG.md is updated
- [ ] Release notes follow format
- [ ] No forbidden files in staging
- [ ] All tests pass
- [ ] Documentation is current
